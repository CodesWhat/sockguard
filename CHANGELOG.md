## [Unreleased]

### Added

- Added an opt-in `POST <admin.path>` (default `/admin/validate`) endpoint that accepts a candidate YAML config in the request body, runs the same parse/validate/compile pipeline as the offline `sockguard validate` command, and returns a structured JSON report (`{"ok": bool, "rules": int, "profiles": int, "compat_active": bool, "errors": [...]?}`). The endpoint is disabled by default (`admin.enabled: false`); when enabled it rides the main listener and inherits the listener's CIDR allowlist, mTLS posture, and per-profile rate-limit / concurrency caps. The handler hard-caps request bodies at `admin.max_body_bytes` (default 512 KiB), returns `405` with `Allow: POST` on non-POST methods, `413` on oversize bodies, `422` with the validation errors on a failing candidate, and `200` on success. Running policy is never mutated. Useful as a CI gate (`curl --data-binary @candidate.yaml http://sockguard/admin/validate`) before promoting a config to production. Companion `config.LoadBytes` helper parses YAML bytes without applying `SOCKGUARD_*` env overlays so the verdict reflects exactly what was submitted.
- Added `admin.path` / `admin.max_body_bytes` config validation: `path` must start with `/` and must not collide with `health.path` or `metrics.path` when those endpoints are also enabled; `max_body_bytes` must be `> 0`.
- Added opt-in hot reload of policy at runtime. When `reload.enabled: true`, sockguard watches the loaded config file via `fsnotify` (Linux inotify / macOS kqueue) and also reloads on `SIGHUP`. Filesystem events are debounced (`reload.debounce_ms`, default `250`) so a single editor save — which commonly emits a burst of chmod/write/rename/create events — collapses into one reload. The reload pipeline parses the new file, applies the same Tecnativa-compat expansion the startup path uses, runs the full validator + rule compiler, and atomically swaps the running handler chain on success. On any failure the running policy is preserved and the failure is logged. A rebuild that would mutate an **immutable** field — `listen.*`, `upstream.socket`, `log.*`, `health.*`, `metrics.*`, or `admin.*` — is refused (those fields are bound to long-lived sockets and goroutines that cannot be replaced without a process restart). Two new metrics surface the outcome: `sockguard_config_reload_total{result="ok|reject_load|reject_validation|reject_immutable"}` counter, and a `sockguard_config_reload_last_success_timestamp_seconds` gauge that is omitted until the first successful reload. SIGHUP behavior changes when `reload.enabled` is true: previously SIGHUP terminated sockguard (Go's default action for unhandled SIGHUP); with reload enabled, SIGHUP triggers a reload and never terminates the process. Default is `reload.enabled: false` for backward compatibility.
- Added policy versioning. The proxy maintains a monotonic generation counter that ticks once at startup and once per successful hot reload; the counter is published two ways. `GET <admin.policy_version_path>` (default `/admin/policy/version`) returns a JSON snapshot — `{"version": int, "loaded_at": RFC3339, "rules": int, "profiles": int, "compat_active": bool, "source": "startup"|"reload", "config_sha256": hex}` — and a new `sockguard_policy_version` Prometheus gauge mirrors the same counter for scrape-side monitoring. The endpoint is opt-in via `admin.enabled: true` (it shares the admin layer with `POST /admin/validate` and inherits the same listener CIDR allowlist, mTLS, and rate-limit posture). The version only ticks on a successful apply, so a stable value across two scrapes means the running policy genuinely did not move — useful for confirming that an operator-issued config change actually took effect. `config_sha256` is a SHA-256 of the effective config's JSON encoding so two snapshots with the same version (always equal) but different content hashes flag a config-drift bug; the field is best-effort and may be empty if the encoder fails. Non-GET methods on the endpoint return `405 Allow: GET`; querying before the first publish returns `503`. Path is validated at config load: must start with `/` and must not collide with `admin.path`, `health.path`, or `metrics.path`.
- Added an optional dedicated admin listener. When `admin.listen.socket` or `admin.listen.address` is configured, the admin endpoints (`POST <admin.path>` and `GET <admin.policy_version_path>`) move to a separate `http.Server` bound to that address — the main Docker-API listener never sees admin traffic, and admin traffic never traverses the Docker-API filter, rate-limit, or client-ACL chain. The dedicated listener supports the same posture as `listen.*`: unix socket (hardened `0600` mode), loopback or TLS-protected TCP (with `admin.listen.tls.cert_file`/`key_file`/`client_ca_file` plus the existing CN/DNS/IP/URI/SPIFFE/SHA-256 SPKI client-identity selectors), or non-loopback plaintext TCP behind an explicit `admin.listen.insecure_allow_plain_tcp=true` opt-in. Validation rejects partial TLS configs, mismatched socket modes, and admin/main address collisions before the proxy starts taking traffic. When `admin.listen` is unset, the existing in-band behavior (admin endpoints share the main listener and inherit its CIDR allowlist, mTLS, and rate-limit posture) is preserved. The admin `http.Server` runs with explicit 30s `ReadTimeout`/`WriteTimeout` since admin endpoints never stream, and unknown paths on the dedicated listener return `404` rather than falling through to the Docker-API filter. An admin Serve failure is treated as fatal: the process exits with a wrapped `admin server error` so operators who opted into a dedicated control plane never silently lose it. The `admin.*` block (including `admin.listen.*`) remains immutable across hot reload — restart is required to change which listener admin rides.

## [0.7.0] - 2026-05-13

### Added

- Added `request_body.container_create.allowed_device_requests` — a structured opt-in for `HostConfig.DeviceRequests` that lets operators allowlist specific device request drivers (e.g. nvidia, amd) with fine-grained capability and count controls, without granting the blanket `allow_device_requests: true` escape hatch. Each allowlist entry specifies a required `driver` field (case-insensitive exact match), an `allowed_capabilities` list of capability sets (each capability set in the request must be a subset of at least one allowlisted set), and an optional `max_count` bound (`-1` = all devices; request `Count: -1` is only permitted when `max_count` is also `-1`). Driver names and capability strings are canonicalized at load time (lowercased, capability sets sorted and deduplicated). Entries with an empty driver are skipped. Malformed requests (empty Driver field) are denied with a 400-style message. If `allow_device_requests: true` is set, the structured allowlist is bypassed entirely (escape hatch wins). If neither is set, all `HostConfig.DeviceRequests` are denied by default (preserving existing behavior).
- Added `clients.profiles[*].limits` for per-profile rate limiting and concurrency caps. `limits.rate.tokens_per_second` and `limits.rate.burst` configure a token-bucket rate limiter; when the bucket is empty sockguard returns `429 Too Many Requests` with `Retry-After: N` and `{"reason":"rate_limit_exceeded","retry_after_seconds":N}`. `limits.concurrency.max_inflight` caps simultaneous in-flight requests per profile; excess requests receive `429` with `{"reason":"concurrency_cap"}` and no retry-after. Either sub-block is optional (omitting both preserves backward-compatible behavior). Validation is fail-closed at startup: `tokens_per_second > 0`, `burst >= tokens_per_second` (or `0` to default to `tokens_per_second`), and `max_inflight > 0`. Anonymous callers (no resolved profile) are bucketed under `_anonymous` so they cannot bypass limits by skipping identification. Added `sockguard_throttle_total{profile,reason}` counter and `sockguard_inflight_requests{profile}` gauge to the Prometheus metrics registry. Throttle audit records are emitted as structured slog events sampled to the first occurrence of each `(profile, reason)` tuple per second to prevent log-volume blowout under load.
- Added `clients.profiles[*].limits.rate.endpoint_costs` — per-endpoint token-cost weighting on top of the base token bucket. Each entry has a `path` glob (same dialect as filter rules, matched against the normalized request path), an optional case-insensitive `methods` list, and a `cost` (number of tokens withdrawn on match; must be `>= 1` and `<= effective burst`). Rules evaluate in declaration order — first match wins — and unmatched requests default to `cost: 1`, preserving prior semantics. Lets operators tax expensive Docker operations (image pull, build, exec) proportionally without lowering the base rate for every endpoint. Throttle audit records emitted by cost-weighted denials carry the `cost` attribute so operators can distinguish expensive-endpoint denials from base-rate denials. Validation is fail-closed at startup: empty paths, malformed globs, costs below `1`, costs exceeding effective burst, and empty method strings all reject the config.
- Added `clients.global_concurrency.max_inflight` plus `clients.profiles[*].limits.priority` (one of `low` / `normal` / `high`, default `normal`) — a system-wide priority-aware fairness gate that prevents noisy low-priority callers from starving higher-priority profiles. When `global_concurrency` is set, each request is admitted only while total in-flight is below its priority share of the global cap (`low=50%`, `normal=80%`, `high=100%`). Per-profile concurrency caps still apply on top of this gate. Denials return `429 Too Many Requests` with `{"reason":"priority_floor"}` and feed a new `priority_floor` reason value into `sockguard_throttle_total` and the sampled slog throttle audit. Profiles without per-profile limits or priority configured are admitted as `normal`, so a single client cannot evade the gate by skipping per-profile config. Validation is fail-closed at startup: `max_inflight > 0` and `priority ∈ {low, normal, high}`.
- Added `request_body.container_create.allowed_device_cgroup_rules` — a structured opt-in for `HostConfig.DeviceCgroupRules` that lets operators allowlist specific device classes (e.g. `c 1:3 rwm` for `/dev/null`, `c 226:* rwm` for DRI/GPU devices) without granting the blanket `allow_device_cgroup_rules: true` escape hatch. Each entry uses Docker's canonical cgroup rule format `<type> <major>:<minor> <perms>` with `*` wildcards for major and/or minor. Entries are canonicalized at load time (whitespace normalized, perms sorted to `r`→`w`→`m` order, duplicates removed). Request rules must canonicalize successfully and match an allowlist entry; wildcards in the request are only permitted when the matching allowlist entry also uses a wildcard at the same position. Malformed request rules are denied with a `400`-style message. The default empty list preserves the existing default-deny for all device cgroup rules.
- Added `response.name_patterns` and `response.image_patterns` glob-based selector axes on the read-side visibility layer. When set, containers and images must also match at least one pattern per configured axis (AND semantics across axes, OR within each axis). `name_patterns` is matched against `Names[0]` (leading `/` stripped) for containers and against each `RepoTags` short name for images; `image_patterns` is matched against the container `Image` field and against each full `RepoTags` reference for images. Both knobs are supported on per-profile visibility policies via `clients.profiles[*].response.name_patterns` / `image_patterns`. Uses the same glob dialect as rule path patterns (`*` = no slash, `**` = any depth).
- Added `examples/compose/` with four standalone Docker Compose stacks (drydock, Traefik, Portainer, Watchtower), each with a self-contained `sockguard.yaml` config and a `README.md` covering audience, exposed surface, and security tradeoffs. Added `chart/sockguard/` — a minimal Helm chart that deploys sockguard as a host-level DaemonSet, mounting `/var/run/docker.sock` read-only and exposing the filtered API on a ClusterIP Service. Chart values expose `image`, `config` (literal YAML for sockguard.yaml), `resources`, `tolerations`, `nodeSelector`, and `affinity`.
- Added the 0.6.0 secure-container enforcement rails on `POST /containers/create`. Each is off by default unless noted as a default-deny rollout below: `request_body.container_create.require_no_new_privileges` (require `HostConfig.SecurityOpt` to include `no-new-privileges:true`), `require_non_root_user` (require non-zero `Config.User`), `require_readonly_rootfs` (require `HostConfig.ReadonlyRootfs=true`), `require_drop_all_capabilities` (require `HostConfig.CapDrop` to contain `"ALL"`), `require_memory_limit` / `require_cpu_limit` / `require_pids_limit` (require positive memory, CPU, and PID budgets), `allowed_seccomp_profiles` + `deny_unconfined_seccomp` (seccomp profile allowlist plus a standalone unconfined kill switch), `allowed_apparmor_profiles` + `deny_unconfined_apparmor`, and `required_labels` (`Config.Labels` keys that must be present with non-empty values).
- Added `request_body.container_create.allow_all_capabilities` and `allowed_capabilities` so operators can govern `HostConfig.CapAdd`. Capabilities are normalised case-insensitively with the `CAP_` prefix optional.
- Added `request_body.container_create.allow_host_userns` so `HostConfig.UsernsMode=host` is denied by default, consistent with the existing host-namespace defaults for network/PID/IPC.
- Added `request_body.container_create.image_trust` — cosign-backed container image signature verification on `POST /containers/create`. Supports keyed verification (PEM-encoded ECDSA/RSA/ed25519 public keys listed under `allowed_signing_keys`) and keyless verification (Fulcio cert chains matched by exact OIDC issuer URL and subject SAN regex listed under `allowed_keyless`). Three modes: `off` (default, skip), `warn` (verify, emit structured audit log on failure, always allow), `enforce` (verify, deny with a clear reason on failure). `require_rekor_inclusion: true` additionally requires a Rekor transparency log entry for keyless bundles. `verify_timeout` controls the per-verification network timeout (default 10s). Configurable per client profile via `clients.profiles[*].request_body.container_create.image_trust`. Backed by `github.com/sigstore/sigstore-go`.

### Changed

- **Default deny for `HostConfig.CapAdd`** when no allowlist is configured. Previous releases left `CapAdd` uninspected on the container-create body inspector. To keep the previous behavior set `request_body.container_create.allow_all_capabilities: true` explicitly, or list the capabilities your workloads need under `allowed_capabilities`.
- **Default deny for `HostConfig.UsernsMode=host`**. Set `request_body.container_create.allow_host_userns: true` to keep `userns_mode: host` working.
- Consolidated the three side-channel Docker socket callers (`internal/ownership`, `internal/clientacl`, `internal/visibility`) into a shared `internal/dockerclient` package so transport configuration is defined in one place.
- Collapsed `*WithDeps`/`*Deps` injection-struct pattern across `filter`, `testcert`, `clientacl`, `ownership`, and `proxy` packages: replaced with package-level `var <name>Hook = <realImpl>` declarations (proxy/filter) or direct function parameters (clientacl/ownership); no public API change.

### Fixed

- Fixed a connection-leak regression in `ownership`, `clientacl`, and `visibility` middleware: all three previously re-dialed the Docker Unix socket on every request because their inline `http.Transport` did not set `MaxIdleConnsPerHost` or `IdleConnTimeout`. They now share the `dockerclient.New` factory, which sets `MaxIdleConnsPerHost: 10` and `IdleConnTimeout: 90s`, matching the main reverse-proxy transport and enabling idle-connection reuse.

## [0.4.0] - 2026-05-04

### Added

- Extended request-body inspection across the remaining guarded write gaps: network create/connect/disconnect, container update/archive, image load, swarm unlock, and node update. These inspectors add bounded JSON/tar reads, request-body preservation for safe requests, endpoint-specific policy options, and startup validation updates so those paths no longer require `insecure_allow_body_blind_writes`.
- Added `clients.client_certificate_profiles[].public_key_sha256_pins` so verified mTLS client certificates can be routed to named profiles by SHA-256 SPKI pin, using the same normalization as listener-level `listen.tls.allowed_public_key_sha256_pins`.
- Added container-create host namespace enforcement for `HostConfig.PidMode=host` and `HostConfig.IpcMode=host`, both denied by default with explicit `request_body.container_create.allow_host_pid` and `allow_host_ipc` opt-ins.
- Added container-create device policy enforcement: `HostConfig.Devices` now requires `request_body.container_create.allowed_devices` or `allow_all_devices`, while `HostConfig.DeviceRequests` and `HostConfig.DeviceCgroupRules` are denied unless their explicit create-time opt-ins are set.

### Changed

- Replaced `-timeout=0` fuzz invocations with finite Go test watchdogs across pre-push, PR, nightly, monthly, and extended-fuzz tiers so a hung fuzz input fails with a goroutine dump instead of tying up the runner until GitHub loses the job. `scripts/local-fuzz.sh` now supports `--timeout`, `--parallel`, and `--suite ultra`, and root-level native runs resolve common Go install paths when `go` is not already on `PATH`.
- Updated README, docs, website comparison copy, and the Drydock preset to reflect the expanded inspection and profile-selector surface. The blind-write guardrail now primarily covers arbitrary exec without an allowlist and plugin setting writes without allowed assignment prefixes.
- Expanded response-side redaction dispatch beyond `GET 200`: protected successful Docker JSON response shapes are now mediated across request methods and body-bearing 2xx statuses, while non-success, no-body, and streaming responses remain outside response rewriting.

### Fixed

- Fixed `POST /build` policy ordering so `networkmode=host` is still denied when remote build contexts and Dockerfile `RUN` instructions are both explicitly allowed.
- `sockguard validate` and `sockguard serve` now fail fast when `--config` explicitly points to a missing file instead of silently continuing with built-in defaults. The absent built-in default path remains allowed when the flag is not provided.
- `sockguard serve`, `sockguard validate`, and `sockguard match` now reject an explicitly empty `--config ""` value during config preflight instead of silently falling through to default config behavior.
- Enabled GitHub private vulnerability reporting, replaced the invalid GitHub noreply disclosure contact with `security@getsockguard.com`, and added `security@getsockguard.com` plus `hello@getsockguard.com` contact links to the issue chooser.
- Cleared local code-scanning findings by refactoring the path-clean fast path away from an off-by-one loop shape, sanitizing benchmark mock-Docker request logs, and scoping the release-cut workflow's write token permission to the tag-push job.
- Fixed the weekly security workflow's Gosec SARIF upload by removing the unsupported `working-directory` input from the Docker action, scanning `./app/...` explicitly, and uploading the SARIF file from the path Gosec actually writes.
- Fixed three `scripts/local-fuzz.sh` and CI workflow regressions introduced alongside the watchdog work: the Docker dry-run branch now passes `--parallel` correctly, `fuzz-duration.sh` no longer rejects a zero-second duration (valid when a caller omits an optional budget), and the nightly/monthly workflow fuzz-budget output steps are now guarded so a missing budget variable does not fail the step before the matrix summary runs. Deduplicated the root `package-lock.json` as a follow-on cleanup.

### Changed

- Removed the fixed-size matching policy buffer from filter inspector selection; the allowed-request path now uses a two-pass severity scan with zero allocations regardless of inspector count.
- Documented that audit events always include the ownership context and repeat the configured `ownership.owner` identifier on every event when ownership is enabled.
- Documented the unavoidable Docker API TOCTOU window between exec metadata inspection and forwarding `POST /exec/*/start`.
- Documented the logging path-field contract for SIEM integrations: access-log `path` and audit-log `raw_path` preserve client-controlled request paths for forensics, while `normalized_path` is the canonical policy path to use for alerting, grouping, and allow/deny analysis.
- Skipped `POST /containers/create` body buffering when the container-create policy is fully permissive (`allow_privileged`, `allow_host_network`, and `/` bind mounts), leaving the original request body untouched for Docker instead of reading up to the 1 MiB inspector cap unnecessarily.
- Removed the per-request temporary slice allocation from filter request-body inspector selection by using a stack-backed matching policy buffer on the allowed-request hot path.
- Moved request-body-to-filter policy option mapping onto config structs via `ToFilterOptions`, so serve and client-profile policy assembly no longer hand-copy every request-body field.
- Threaded the configured listener type into audit logging instead of inferring Unix listeners from an empty request `RemoteAddr`. Audit events now get `transport_listener` from serve listener configuration through `AuditOptions`, while request address parsing remains limited to actor identity fields.
- Extracted shared bind-source allowlist matching into `bindPathAllowed`, removing the duplicated root/exact/child-path checks from container-create, service, and plugin bind-mount policy paths.
- Reused the shared bounded-body reader across exec-create, container-create, plugin privilege/set, service, and swarm inspectors. These paths now all get the same `ContentLength` fast-path rejection, restored downstream request bodies, and post-read close-error handling as the existing volume/secret/config inspectors.
- Moved dedicated audit-log writes off the request hot path and trimmed per-request audit allocations. Audit events now enqueue into a bounded channel and are encoded by a single background writer goroutine, so a slow audit sink no longer serializes request completion behind a global mutex; shutdown drains queued events before closing the configured audit output. Actor and transport identity fields are now flattened strings, and the static ownership context is precomputed when the middleware is built.
- `inspectAllowedRequest` now narrows request-body inspection to the highest-severity matching endpoint policy instead of walking every inspector in registration order. That means `POST /containers/create`, `/build`, service/swarm writes, plugin writes, and the smaller bounded-body inspectors dispatch directly to their relevant matcher, avoiding irrelevant policy calls once the applicable inspection tier is known.
- Client ACL profile selection now logs the winning resolution strategy at debug level. When a request is assigned a per-client profile, Sockguard emits `client ACL profile matched` with the selected profile name plus whether it came from a client certificate, unix peer credentials, source IP, or the configured default profile.
- Indexed repeated client profile lookups in `internal/clientacl` by concrete source IP and client-certificate fingerprint. Sockguard still evaluates source-IP CIDRs and certificate selector assignments in config order on the first sighting of a caller, but repeated requests from the same IP or the same parsed mTLS leaf now reuse a memoized first-match-wins result instead of linearly rescanning every configured profile assignment on every request.
- Unified `POST /containers/create` bind-source validation so classic `HostConfig.Binds` entries and `HostConfig.Mounts` bind entries now flow through the same `extractAndValidateBindSource` helper before allowlist checks. This removes the duplicated path-parsing branch in `internal/filter/container_create.go` and keeps future bind-source validation changes aligned across both Docker request shapes.
- Decomposed `internal/cmd/serve.go`'s handler assembly into named `withHijack`, `withOwnership`, `withVisibility`, `withFilter`, `withHealth`, `withClientACL`, `withRequestID`, and `withAccessLog` middleware layers that are chained in order from a small loop. The serve path keeps the same middleware ordering and behavior, but the composition is now explicit, unit-tested, and no longer buried inside one long handler-construction function.

### Security

- Rebuilt hijack upstream request targets from Sockguard's normalized Docker path, a fixed `docker` host, and canonicalized query parameters instead of copying caller-supplied URL and `Host` fields verbatim to the Docker socket.
- Client-certificate profile selection now ignores unverified `tls.ConnectionState.PeerCertificates` entries and only matches against the verified mTLS leaf from `VerifiedChains`, preventing a misconfigured listener from assigning privileged profiles based on a caller-supplied certificate subject alone.

This closes the easy misconfiguration path where a too-broad mode plus umask math could leave the proxy socket readable or writable by unintended local users.

- Expanded read-side visibility control. `response.visible_resource_labels` and `clients.profiles[].response.visible_resource_labels` now inject Docker label selectors into service/task/secret/config/node lists in addition to the existing container/image/network/volume/event coverage, hide hidden inspect/log-style reads for those resource families behind `404`, hide `GET /swarm` when its labels do not match, and apply additively with selected client profiles.
- Completed request-aware enforcement for the remaining non-Swarm high-risk write paths. Sockguard now inspects `POST /containers/*/exec` bodies against an argv allowlist plus privileged/root-user toggles, re-inspects `POST /exec/*/start` via upstream `GET /exec/{id}/json` before execution, inspects `POST /images/create` query parameters to block `fromSrc` imports and constrain registries, and inspects `POST /build` to block remote contexts, `networkmode=host`, and Dockerfiles containing `RUN` instructions unless those behaviors are explicitly allowed. `/build` inspection replays the request body from a temp file so local tar contexts still reach Docker unchanged after policy checks.

Before, every inbound request from an allowed CIDR fired a fresh upstream Docker API call to map source IP → caller container; a burst of N concurrent requests amplified into N upstream calls, turning one allowlisted client into a steady source of Docker-daemon load. The new setup escalates coverage across four tiers, matrix-parallelized so every tier's wall time is near its per-fuzzer budget regardless of how many fuzzers we ship:

Upload retention 180 days so a regression found by a monthly run stays recoverable long enough to turn into a committed seed.

### Fixed
